# Hero3 MVP 设计文档

最后更新：2026-05-20

## 1. 项目定位

Hero3 是一个三国题材策略网页游戏。第一版目标不是做完整 SLG，而是先做出可持续扩展的核心闭环：

```text
资源生产 -> 城池建设 -> 征兵养成 -> 地图行动 -> 战斗结算 -> 奖励回流
```

项目采用前后端分离结构：

- `web/`：React + TypeScript + Vite，负责页面展示和玩家交互。
- `go/`：Go API 服务，负责核心数据、规则结算、存档和后续服务端逻辑。

`webgame_wlsg` 作为玩法结构参考，但 Hero3 不直接继承它的前端本地状态架构。

## 2. MVP 目标

MVP 要证明三件事：

1. 玩家能完成一轮完整成长循环。
2. 核心数据以后端为准，刷新页面后状态仍然一致。
3. 城池、军事、地图、战报四个模块之间的数据能够闭环。
4. 同一套前端页面能兼容桌面端和手机端，不为移动端另做一套割裂页面。

第一版成功标准：

- 玩家可以创建基础角色。
- 玩家拥有资源、建筑、军队和地图目标。
- 资源能随时间恢复或增长。
- 玩家可以升级建筑。
- 玩家可以征兵。
- 玩家可以攻击地图目标。
- 后端能返回战斗结果和奖励。
- 前端能展示当前状态和战报。
- 手机端能完成创建玩家、查看资源、升级建筑、征兵、攻击目标和查看战报。

## 3. 非目标

以下内容不进入第一版 MVP：

- 联盟、公会、多人协作。
- 实时 PVP。
- 排行榜。
- 复杂武将抽卡与养成。
- 邮件系统。
- 每日公告。
- 付费系统。
- 复杂地图行军和驻军。
- 多服务器分区。
- 管理后台。

这些系统可以在核心闭环稳定后再做。

## 4. 核心玩法循环

第一版主循环：

```text
进入游戏
-> 创建玩家
-> 获取游戏状态
-> 资源随时间增长
-> 升级资源建筑或军事建筑
-> 消耗资源征兵
-> 选择地图目标
-> 发起攻击
-> 后端计算胜负、损耗和奖励
-> 生成战报
-> 玩家获得资源或成长
-> 回到城池继续发展
```

循环中的核心决策：

- 先升级资源建筑，还是先征兵。
- 攻击低等级目标稳定拿收益，还是挑战高等级目标。
- 资源用于建筑升级，还是转化为军队战力。

## 5. 数据归属原则

前端负责：

- 页面布局和组件状态。
- 表单输入、弹窗、菜单、标签页。
- 倒计时展示。
- 接口 loading、错误提示、空状态。
- 从后端状态派生出的展示数据。

后端负责：

- 玩家身份。
- 资源数量。
- 建筑等级和升级进度。
- 军队数量。
- 征兵队列。
- 地图目标。
- 战斗结算。
- 奖励发放。
- 战报记录。
- 时间结算和离线恢复。

关键原则：

- 前端不能自行决定资源增长、战斗胜负、奖励数量。
- 玩家操作必须通过后端动作接口完成。
- 前端每次进入游戏时通过 `GET /api/v1/game/state` 获取权威状态。
- 动作接口完成后应返回新的权威状态或足够前端更新的结果。

## 6. 第一版页面

页面需要从第一版开始兼容手机端展示。桌面端可以使用侧边导航和多列布局，手机端应改为底部导航、单列信息流和可折叠详情，保证核心操作不依赖悬停、不依赖过宽表格。

### 6.1 城池页

用途：

- 展示玩家资源。
- 展示建筑列表。
- 发起建筑升级。
- 展示升级倒计时。

第一版建筑：

- 木场：提升木材产出。
- 采石场：提升石料产出。
- 铁矿：提升铁矿产出。
- 农田：提升粮食产出。
- 兵营：解锁或提升征兵能力。
- 仓库：提升资源容量。

### 6.2 军事页

用途：

- 展示当前军队。
- 展示可训练兵种。
- 发起征兵。
- 展示征兵队列。

第一版兵种：

- 步兵：基础肉盾，成本低。
- 弓兵：输出单位，消耗木材和粮食。
- 骑兵：高战力单位，消耗铁矿和粮食。

### 6.3 地图页

用途：

- 展示可攻击目标。
- 查看目标等级、守军和奖励预览。
- 发起攻击。

第一版目标：

- 流寇营地：低难度，资源奖励少。
- 山寨据点：中等难度，资源奖励中等。
- 叛军营地：高难度，资源奖励较高。

### 6.4 战报页

用途：

- 展示最近战斗记录。
- 展示胜负、我方损失、敌方损失、奖励。
- 作为地图行为的反馈闭环。

## 7. 第一版数据模型

### 7.1 Player

```text
id
nickname
faction
createdAt
updatedAt
```

说明：

- `faction` 第一版可以先保留字段，不急着做复杂阵营加成。

### 7.2 ResourceState

```text
playerId
wood
stone
iron
food
gold
capacity
lastCalculatedAt
```

说明：

- 资源增长由后端根据 `lastCalculatedAt` 结算。
- `capacity` 可以由仓库等级动态计算，也可以在响应中作为派生字段返回。

### 7.3 Building

```text
id
playerId
type
level
upgradeStartedAt
upgradeEndsAt
```

说明：

- `upgradeEndsAt` 为空表示当前没有升级。
- 第一版同类型建筑可以只有一座，后续再扩展多建筑位。

### 7.4 ArmyUnit

```text
playerId
unitType
amount
```

说明：

- 军队数量以后端为准。
- 战斗损耗由后端直接扣减。

### 7.5 RecruitQueue

```text
id
playerId
unitType
amount
startedAt
endsAt
status
```

说明：

- `status` 可用 `pending`、`completed`、`claimed`。
- 第一版可以在获取游戏状态时自动结算已完成队列。

### 7.6 MapTarget

```text
id
type
level
power
rewardWood
rewardStone
rewardIron
rewardFood
expiresAt
```

说明：

- 第一版可以使用服务端固定生成的目标列表。
- 后续再做玩家独立地图、刷新规则和坐标。

### 7.7 BattleReport

```text
id
playerId
targetId
result
playerPower
enemyPower
lostUnits
rewards
createdAt
```

说明：

- 战报只记录结算结果，不作为实时战斗回放。

## 8. 第一版接口

### 8.1 玩家

创建玩家：

```text
POST /api/v1/player/create
```

请求：

```json
{
  "nickname": "主公",
  "faction": "wei"
}
```

返回：

```json
{
  "playerId": "player_001",
  "state": {}
}
```

### 8.2 游戏状态

获取权威游戏状态：

```text
GET /api/v1/game/state?playerId=player_001
```

返回应包含：

```text
player
resources
buildings
army
recruitQueues
mapTargets
recentBattleReports
serverTime
```

### 8.3 城池

升级建筑：

```text
POST /api/v1/city/buildings/upgrade
```

请求：

```json
{
  "playerId": "player_001",
  "buildingType": "wood_camp"
}
```

返回：

```text
更新后的资源、建筑和 serverTime
```

### 8.4 军事

发起征兵：

```text
POST /api/v1/military/recruit
```

请求：

```json
{
  "playerId": "player_001",
  "unitType": "infantry",
  "amount": 10
}
```

返回：

```text
更新后的资源、军队、征兵队列和 serverTime
```

### 8.5 地图与战斗

攻击地图目标：

```text
POST /api/v1/map/targets/attack
```

请求：

```json
{
  "playerId": "player_001",
  "targetId": "target_001",
  "units": {
    "infantry": 20,
    "archer": 10
  }
}
```

返回：

```text
battleReport
updatedResources
updatedArmy
updatedMapTargets
```

### 8.6 战报

获取战报：

```text
GET /api/v1/battle/reports?playerId=player_001
```

返回：

```text
reports[]
```

## 9. 第一版数值规则

以下数值先作为占位，后续可以按你的具体规则调整。

资源：

- 木材、石料、铁矿、粮食为基础资源。
- 金币先保留，不进入第一版核心循环。
- 每种资源都有基础产出和仓库容量限制。
- 后续可以增加“金币消费的限时总产量加成”模块。该模块独立于建筑配置、阵营加成和资源地基础产量，属于玩家个人状态。
- 总产量加成只影响最终产量公式：`最终资源产量 = 基础资源地产量合计 × 总产量倍率`。
- 加成需要记录倍率、开始时间、结束时间和来源，例如 `source = coin`。第一版资源模块不实现该功能，只保留后续接入点。
- 因为加成有时间限制，后续实现离线结算时需要处理跨过期时间的结算切片，不能把过期后的离线时间也按加成倍率计算。

建筑：

- 建筑升级消耗资源。
- 建筑升级需要时间。
- 资源建筑提升对应资源产出。
- 仓库提升资源容量。
- 兵营影响可训练兵种或征兵速度。

征兵：

- 征兵消耗资源。
- 征兵需要时间。
- 征兵完成后增加军队数量。

战斗：

- 第一版采用简化战力结算。
- 玩家战力由出征兵种和数量计算。
- 敌方战力由目标等级和类型计算。
- 胜利获得奖励，失败不获得奖励。
- 无论胜负都可能产生兵力损失。

## 10. 前端状态策略

前端可以有本地状态，但只保存 UI 层内容：

- 当前页面。
- 当前标签页。
- 弹窗开关。
- 表单输入。
- loading 状态。
- 错误提示。

前端不持久保存核心游戏数据。核心状态来自后端：

```text
GET /api/v1/game/state
```

建议前端创建一个 `gameApi` 和一个全局 `gameState`：

- `gameApi`：封装所有后端请求。
- `gameState`：保存当前后端返回的权威状态。
- 页面组件只消费 `gameState` 派生出的展示数据。

## 11. 代码规模规范

为了避免页面文件变成难维护的大文件，Hero3 从 MVP 阶段开始限制单文件规模。

### 11.1 前端页面文件

React 页面文件建议控制在：

```text
150 行以内
```

硬性上限：

```text
220 行
```

超过 `150` 行时，应优先拆分：

- 页面内区域组件，例如 `ResourcePanel`、`BuildingList`、`RecruitQueue`。
- 纯展示组件，例如 `StatCard`、`ActionButton`、`ReportItem`。
- 页面逻辑 hook，例如 `useCityPage`、`useRecruitForm`。
- 配置数据，例如建筑列表、资源展示项、导航项。

超过 `220` 行时，除临时验证代码外，不应继续追加功能，应先重构拆分。

### 11.2 前端组件文件

普通组件建议控制在：

```text
120 行以内
```

硬性上限：

```text
180 行
```

组件如果同时包含复杂布局、数据转换、事件编排和状态管理，应拆为展示组件、hook 和工具函数。

### 11.3 Go 文件

Go 文件建议控制在：

```text
200 行以内
```

硬性上限：

```text
300 行
```

超过建议线时，应按职责拆分：

- `handler` 只做 HTTP 参数和响应。
- `service` 放业务流程。
- `rules` 或 `calculator` 放资源、建筑、征兵、战斗等规则计算。
- `store` 放数据读写。

### 11.4 通用原则

- 单个函数建议控制在 `50` 行以内。
- 一个文件只承担一个清晰职责。
- 不为了行数机械拆分，但当页面开始混合 UI、状态、接口、规则时必须拆。
- 新功能优先新增小组件、小 hook、小服务，而不是继续堆到页面文件。

## 12. 移动端适配要求

Hero3 第一版必须把手机端作为正式目标，而不是后期补丁。移动端适配以竖屏手机为主，同时保证桌面端信息密度。

### 12.1 目标设备

优先适配：

- 竖屏手机：`360px` 到 `430px` 宽度。
- 平板或窄窗口：`768px` 左右宽度。
- 桌面端：`1280px` 及以上宽度。

最低验收宽度：

```text
360px
```

### 12.2 导航结构

桌面端：

- 可以使用左侧导航或顶部导航。
- 主内容允许两列或三列布局。
- 资源、队列、战报摘要可以同时展示。

手机端：

- 使用底部导航承载主要页面：城池、军事、地图、战报。
- 顶部只保留玩家名、资源摘要或当前页面标题。
- 次级信息放入折叠面板、抽屉或详情弹窗。
- 关键操作按钮固定在可触达区域，避免放在过深页面底部。

### 12.3 页面布局

城池页：

- 手机端资源条横向可换行或压缩为两行。
- 建筑列表使用单列卡片。
- 升级按钮和倒计时必须在卡片首屏可见。

军事页：

- 兵种列表使用单列。
- 征兵数量使用步进器、滑条或数字输入，不能只依赖键盘精确输入。
- 征兵队列放在当前页下方或独立折叠区。

地图页：

- 目标列表使用单列。
- 目标详情可以用底部抽屉展示。
- 攻击前必须清楚展示预计风险、派兵数量和奖励。

战报页：

- 战报列表使用可扫描的摘要卡片。
- 详细损失和奖励放入展开区域。

### 12.4 交互要求

- 所有可点击目标建议不小于 `44px` 高度。
- 不使用只能靠鼠标悬停查看的关键信息。
- 长文本必须自动换行，不能撑破容器。
- 表格在手机端要改成卡片或分组列表。
- 弹窗在手机端优先使用底部抽屉或全屏面板。
- loading、失败、空状态必须在手机端可读。

### 12.5 视觉密度

Hero3 是策略经营游戏，不做营销落地页式大首屏。移动端应保持信息可扫读：

- 少用大面积装饰图。
- 不把页面主体做成嵌套卡片。
- 优先展示资源、队列、可操作事项。
- 卡片圆角控制在 `8px` 以内，保持工具型界面气质。

### 12.6 技术约定

前端实现时应优先使用：

- CSS Grid 和 Flexbox。
- `minmax()`、响应式网格和明确的 `min-width` / `max-width` 约束。
- 明确的断点，例如 `480px`、`768px`、`1024px`。
- 安全区适配：`env(safe-area-inset-bottom)`。

避免：

- 用视口宽度直接缩放字体。
- 固定宽度表格。
- 只适合桌面的悬浮工具栏。
- 在手机端横向溢出主体内容。

## 13. 后端模块边界

建议 Go 后端按领域拆分：

```text
internal/player
internal/game
internal/city
internal/resource
internal/military
internal/map
internal/combat
internal/report
internal/store
```

职责：

- `player`：玩家创建和基础资料。
- `game`：聚合游戏状态。
- `city`：建筑升级。
- `resource`：资源结算和容量。
- `military`：征兵、军队数量。
- `map`：地图目标。
- `combat`：战斗结算。
- `report`：战报记录。
- `store`：数据库访问。

HTTP 层只做：

- 参数解析。
- 调用业务服务。
- 返回 JSON。

不要把玩法规则直接写在 handler 里。

## 14. 开发顺序

建议按以下顺序实现：

1. 完成 MVP 文档并持续维护。
2. 定义 Go 领域模型。
3. 做内存版 `game/state`，先不接数据库。
4. 前端接入 `game/state`，搭建兼容桌面端和手机端的四个主页面。
5. 实现资源结算。
6. 实现建筑升级。
7. 实现征兵。
8. 实现地图目标。
9. 实现简化战斗。
10. 实现战报。
11. 再接 PostgreSQL 持久化。
12. 在每个页面完成后做桌面端和手机端视觉验收，不等到最后统一补响应式。

说明：

- 先做内存版可以更快验证主循环。
- 数据模型稳定后再接数据库，避免过早迁移反复修改。

## 15. 测试策略

后端优先测试：

- 资源随时间结算。
- 建筑升级消耗和完成。
- 征兵消耗和完成。
- 战斗胜负和损耗。
- 战报生成。

前端优先测试：

- 首页能加载游戏状态。
- 建筑升级按钮状态正确。
- 征兵表单能提交。
- 地图攻击后能看到战报。
- `360px` 宽度下四个主页面无横向溢出。
- 手机端底部导航可用，关键按钮可点击。
- 桌面端和手机端都能完成 MVP 主循环。

第一版至少保证：

```text
go test ./...
pnpm build
```

在每次重要改动后可以通过。

前端 UI 改动还应至少检查：

```text
桌面端：1280x720
手机端：390x844
最低宽度：360x800
```

## 16. 待确认问题

这些问题需要后续逐步确定：

- 是否需要账号登录，还是先使用本地游客 ID。
- 第一版阵营是否产生数值加成。
- 金币限时总产量加成的倍率、持续时间、金币价格和是否允许续时。
- 第一版是否允许多个城池。
- 战斗是否只做即时结算，还是需要行军时间。
- 地图目标是全服共享，还是每个玩家独立生成。
- 资源是否需要离线收益上限。
- 征兵队列是否允许多个并行。
- 战报保留多少条。

- 手机端是否需要保留桌面端同等信息密度，还是优先简化操作。
