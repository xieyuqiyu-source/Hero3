# Hero3 MVP 设计文档 v2

最后更新：2026-05-23

## 1. 项目定位

Hero3 是一个三国题材策略网页游戏。核心目标是做出可持续扩展的成长闭环：

```text
资源生产 → 城池建设 → 征兵养成 → 地图行动 → 战斗结算 → 奖励回流 → 将领成长
```

技术架构：
- `web/`：React + TypeScript + Vite（玩家前端）
- `admin/`：React + TypeScript + Vite（GM 后台）
- `go/`：Go API 服务（核心逻辑、数据结算、存档）
- 数据库：MySQL（持久化）
- 配置驱动：balance.json / factions.json / units/*.json

## 3. 待开发模块（按优先级排序）

### 3.1 PvE 战斗系统

#### 战斗模式
- 数值比拼，不做兵种克制
- 战报类型展示（详细的攻防过程记录）
- 攻防规则由独立配置驱动（后续提供详细规则）

#### 地图目标（NPC）
- 按玩家实力动态生成（建筑总等级/兵力决定难度范围）
- 奖励在一定范围内随机
- 打完有冷却时间（不能无限刷）
- NPC 有具体兵种配置（步兵/骑兵等混编）
- 奖励有上限，防止无限刷资源
- 掉落物：资源 + 将领经验 + 装备/材料

#### 行军时间
- PvE 行军时间较短（1-5 分钟）
- PvP 行军时间取决于出征部队中**最慢兵种的速度**（木桶效应）
- 全骑兵闪电战：行军快，对方反应时间短
- 混编带攻城车：行军慢，对方有时间准备
- 驿站建筑加成行军速度

#### 全服公共据点 — 沉冤池
- 全服共享高难度 NPC 据点，每天刷新一次
- 只有实力达标的玩家才能挑战
- 所有参与者的战损（按口粮计算）累积到"沉冤池"
- 据点被打下后触发奖励结算

**沉冤池奖励机制（B+C 方案）：**
- 个人阶梯保底：累计战损达到门槛即可领取对应奖励
  - 1000 口粮 → 铜箱（基础资源）
  - 5000 口粮 → 银箱（中级材料）
  - 20000 口粮 → 金箱（稀有掉落）
- 全服池子越大，最终击杀者的额外奖励品质越高
  - 池子 < 10000：普通掉落
  - 池子 10000-50000：稀有掉落概率 +20%
  - 池子 > 50000：必出传说级物品
- 所有贡献过战损的人拿参与奖，击杀者拿额外大奖

### 3.2 千帐营 / 口粮系统

#### 核心机制
- 新增特殊建筑"千帐营"，位于资源建筑下方加成区域
- 千帐营等级决定口粮上限
- 口粮消耗 = 所有兵的 upkeep 之和
- 口粮消耗 ≥ 上限时无法征兵
- 将领可以加成千帐营容量

#### 口粮消耗参考
- 步兵：2/个
- 侦察兵：1/个
- 骑兵：3-4/个
- 攻城车：3-4/个
- 运输兵：5/个
- 贵族兵：6/个
- 商人：0

#### 示例
- 千帐营 Lv.1 → 口粮上限 1000 → 最多养 500 步兵
- 千帐营 Lv.5 → 口粮上限 5000 → 最多养 2500 步兵
- 混编（200步 + 100骑 + 50攻城）：消耗 200×2 + 100×3 + 50×3 = 850

### 3.3 将领系统

#### 第一阶段（战斗系统完成后）
- 每个存档初始带一个将领（创建时选的武将）
- 将领只有等级（战斗获得经验）
- 等级提供固定加成（攻+X%、防+X%、建筑速度+X%）
- 加成通过 `map[string]float64` 的 buffs 实现
- 不做加点、不做背包、不做装备

#### 第二阶段
- 加点系统（多维度属性：武力/智力/政治/统率）
- 背包系统（战斗掉落物品存入将领背包）
- 装备系统（装备提供额外 buffs）
- 背包可拆出：资源、加成装备、材料

#### 第三阶段
- 将领特殊技能（按国家/将领不同）
- 跨存档加成（同账号不同存档的将领提供全局加成）
- 将领获取途径（战斗俘虏、招募、活动获取）

#### 数据结构演进
```json
// 第一阶段
{ "id": "guanyu", "name": "关羽", "level": 5, "exp": 1200, "buffs": { "attackBonus": 1.15 } }

// 第二阶段
{ "attributes": { "武力": 8, "智力": 3 }, "buffs": { "attackBonus": 1.15, "recruitSpeedBonus": 1.1 } }

// 第三阶段
{ "inventory": [...], "equipment": { "weapon": "item_001" }, "skills": [...] }
```

### 3.4 PvP 系统（将领第一阶段完成后）

#### 核心机制
- 玩家可以攻击其他玩家的城池
- 行军时间 = 基础时间 / 最慢兵种速度（木桶效应）
- 胜利可掠夺对方资源（受负重限制）
- 失败损失兵力

#### 防御机制
- 城墙：提高防御值
- 烽火台：决定敌方需要消灭你多少比例的兵力才能看到你的真实守军数据
  - Lv.1：敌方需消灭你 5% 兵力才能看到全部守军信息
  - Lv.5：需消灭 25%
  - Lv.10：需消灭 50%
  - 未达到比例时，敌方战报中不显示守方真实兵力
  - 攻方可以选择"侦查"（小规模试探）或"盲打"（全力进攻但看不到对方底牌）
- 驻军防守

#### 增援机制
- 任何玩家都可以增援其他玩家（不限联盟）
- 联盟成员增援有额外加成（行军加速/驻防上限提高）
- 同账号不同存档可以互相增援（后期可调整限制）

### 3.5 阵营特色系统

每个阵营拥有一个**金色专属建筑**，提供独特的正循环玩法：

#### 魏国 — 建设兵团
- 金色建筑，驻军提供资源加成
- 兵越多 → 资源产出越高 → 养更多兵
- 正循环：养兵就是赚钱
- 受千帐营上限限制（不会无限滚雪球）

#### 蜀国 — 桃园
- 金色建筑，结义系统
- 增援无行军时间（义兄弟瞬间到场防守）
- 义兄弟活跃时你也获得成长回馈：
  - 兄弟升建筑 → 你获得 10% 资源回馈
  - 兄弟征兵 → 你免费获得少量同类型兵
  - 兄弟打赢 NPC → 你获得小额奖励
- 正循环：社交就是实力，兄弟越活跃我越强
- 最多结义 2-3 人

#### 吴国 — 转运司
- 金色建筑，资源自动转化
- 每隔一段时间触发"转运"：最多的资源溢出部分按比例转化为金币
- 金币可购买加速、产量加成、特殊道具
- 等级越高转化比例越好
- 正循环：产出永不浪费，爆仓反而在赚钱

#### 阵营特色对比

| 国家 | 金色建筑 | 正循环 | 一句话 |
|------|---------|--------|--------|
| 魏 | 建设兵团 | 兵多→产资源→更多兵 | 养兵就是赚钱 |
| 蜀 | 桃园 | 兄弟活跃→我也变强 | 社交就是实力 |
| 吴 | 转运司 | 爆仓→变金币→买加速 | 产出永不浪费 |

### 3.7 多存档系统

#### 核心定位
- 多存档 = 经营多个城池，鼓励用户多开游玩
- 每个存档因将领不同，核心玩法策略完全不同
- 不是"重复体验"，是"不同体验"

#### 将领决定玩法风格
- 存档 A 选诸葛亮 → 种田流，资源加成爆炸，慢慢发育碾压
- 存档 B 选甄宓 → 劝降流，战斗中夺取敌方兵力为己用
- 存档 C 选徐骁 → 闪电战流，骁骑营极速行军，打完就跑
- 每个将领有独特的被动特性 + 主动技能 + 专属加成

#### 将领设计原则
- 将领 = 玩法定义者，不只是数值加成
- 每个将领有：
  - **被动特性**：永久生效的独特机制（如诸葛亮：所有资源产出 +20%）
  - **主动技能**：战斗/经营中可触发的能力（如甄宓：战后劝降 10% 敌方残兵）
  - **专属加成**：只对特定兵种/建筑/资源生效（如徐骁：骁骑营行军速度 ×2）
- 不同将领引导完全不同的发展路线和战斗策略

#### 跨存档联动
- 同账号不同存档可以互相增援（多城池协防）
- 跨存档声望：所有存档战绩累积到账号级别，解锁全局加成
- 将领池账号共享：A 存档练出的将领可借调到 B 存档（同一时间只能激活一个存档）
- 跨存档资源运输（有损耗和冷却限制）

- 产量加成系统（金币购买 ×2/×4/×8/×16 限时加成）
- 开拓新田地接口
- 功能建筑建造（目前只有升级，没有新建）
- 联盟系统
- 排行榜

## 5. 技术架构要点

### 5.1 配置驱动
- 所有数值（建筑/兵种/将领/战斗）通过 JSON 配置
- bootstrap 接口一次性下发所有配置
- 前端不维护配置副本，从 configStore 读取
- 改数值只改配置文件，不改代码

### 5.2 加成系统统一架构
- 所有加成源（阵营/将领/装备/词条/建筑）共用 `map[string]float64`
- 统一计算函数：`ApplyTraits(base, key, ...sources)`
- 加成是"读配置 → 算结果 → 返回"，不修改原始配置

### 5.3 惰性结算
- 资源产出：按时间差懒结算
- 建筑升级：到期自动 level++（时间切片）
- 征兵队列：到期自动加入 army
- 不需要定时任务，所有结算在 GetState 时触发

### 5.4 数据存储
- 玩家状态整体存为 state_json（灵活，改模型不迁移）
- 配置文件按领域分文件（balance/factions/units）
- 后续加将领/装备只是往 state_json 里加字段

## 6. 安全与上线前必做事项

### 6.1 身份验证（上线前必须）

当前状态：API 只靠前端传 playerId，没有 token/session 验证。

风险：知道别人 playerId 可以操作对方存档（升级、征兵、删除）。

修复方案：登录后发 JWT token，每次请求带 token，后端验证 token 对应的 accountId 是否拥有该 playerId。

时机：PvP 上线前必须完成。

### 6.2 测试接口清理（上线前必须）

以下接口仅供开发测试，上线前必须删除或加 GM 权限：
- `POST /api/v1/city/resources/fill`（一键爆仓，无任何限制）

### 6.3 频率限制（上线前应做）

加 rate limiter，每个 IP/账号每秒最多 N 次请求，防止脚本刷接口。

### 6.4 当前接口安全状态

| 接口 | 后端校验 | 安全 |
|------|---------|------|
| 升级建筑 | 资源/升级中/满级 | ✅ |
| 一键升级 | 逐个检查资源 | ✅ |
| 征兵 | 资源/队列上限/数量上限 | ✅ |
| 爆仓 | 无限制 | ❌ 上线前删除 |
| 删除存档 | 只检查存在 | ⚠️ 需加身份验证 |
| 删除账号 | 只检查存在 | ⚠️ 需加身份验证 |
| 其他读接口 | 无写操作 | ✅ |

### 6.5 幂等性（可后做）

网络抖动可能导致同一请求重复提交。后续可加请求唯一 ID 防重复执行。
