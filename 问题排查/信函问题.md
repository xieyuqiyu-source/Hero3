## 信函系统审核报告

---

### 一、隐藏 Bug

**1. 时间格式不一致，存储和读取会互相踩踏**

`service_mail.go` 里用 `resourceDateLayout` 格式（很可能是自定义格式，如 `2006-01-02T15:04:05Z`）存 `CreatedAt` / `ReadAt`，但 MySQL 层 `scanMail` 回读时是用 `time.RFC3339` 格式输出。如果 `resourceDateLayout` 和 `RFC3339` 不完全一致（比如少了纳秒、时区后缀），内存模式和 MySQL 模式在序列化后返回的时间字符串会不一样，前端 `new Date(mail.createdAt)` 解析一致性无法保证。

**2. MemoryRepository.ListMails 的 total 计算不准**

当前实现里 `total` 是通过遍历时累加算出来的，但这和 SQL 里先 `COUNT(*)` 再查 `LIMIT/OFFSET` 是不同语义。内存版里 `total` 是"分页前可见条数"，而 MySQL 版也是这样——两者逻辑相同，但是 Memory 版里 `offset` 会被消耗掉：它在循环里边减 `offset` 边累加 `total`，导致 `total` 的计数从第一条开始，不论 `offset` 多少。这个逻辑实际是对的，但非常容易在以后改动中出错，且不直观，建议拆成两次遍历。

**3. DeleteMail 没有权限二次校验**

`service_mail.go` 的 `DeleteMail` 直接调 `repo.DeleteMail(playerID, mailID)`，依赖数据库 `WHERE id=? AND player_id=?` 兜底。但 MemoryRepository 的实现是先找 `r.mails[playerID]`，确实有隐性归属校验。然而如果未来有人直接绕 Service 调 Repo，这层语义就消失了。相比 `GetMail` 里有显式的 `mail.PlayerID != playerID` 检查，`DeleteMail` 的防御性更弱。

**4. AdminSendMail 强制覆盖 SenderName**

`handler.go` 里 `AdminSendMail` 写死了 `payload.SenderName = "Hero3 GM"`，但 `SendMailRequest` 里 `SenderName` 是可由调用方传入的。这意味着即使 GM 后台将来想用不同的发件人名字（比如"系统运维"、"活动组"），handler 会把它抹掉。逻辑应该在 service 层判断，不应在 handler 层强制赋值。

**5. 附件 `attachmentsJSON` 存 `null` 和空数组的处理**

`SaveMail` 里 `json.Marshal(mail.Attachments)` 当 `Attachments` 为 `nil` 时会序列化成 `null`，写入 DB 是 NULL。读取时 `scanMail` 检查 `len(attachmentsJSON) > 0` 才反序列化，如果是 `null` 字节不为空（JSON null = 4字节），实际上会触发 `json.Unmarshal([]byte("null"), &mail.Attachments)`，结果是 `nil` slice，行为是正确的但代码路径隐晦。空数组 `[]` 序列化成 `[]byte("[]")`，反序列化后是空 slice，没有问题。这个地方代码没有 Bug，但逻辑不透明，容易被误判。

---

### 二、扩展性局限

**1. MarkMailRead 没有纳入 Repository 接口的统一签名**

设计文档里 `Repository` interface 的 `MarkMailRead` 签名是 `MarkMailRead(playerID string, mailID string) error`（不带 `readAt`），但实际代码里接口和实现都带了 `readAt time.Time`。这是文档和实现已经不同步的第一处。未来维护时容易混淆。

**2. 全局 UnreadMailCount 每次 ListMails 都独立查一次数据库**

`service_mail.go` 的 `ListMails` 每次都调 `repo.CountUnreadMails`，这是两条独立 SQL（MySQL 版），没有在同一事务里执行，存在读一致性问题：列表和未读数可能是两个时间点的快照。数据量小时无感知，高并发或批量操作时会出现未读数和实际列表不一致。

**3. 删除后强制返回第 1 页第 10 条是硬编码**

`DeleteMail` 服务方法最后调 `s.ListMails(playerID, 1, 10)` 返回删除后的列表。这意味着前端如果在第 3 页删了一封信，删完后被强制跳回第 1 页。这个行为是强迫前端接受"删除后重置分页"的设计，未来改成删后停留当前页时，要改 service 签名，破坏调用方。

**4. 附件类型完全没有服务端校验**

`buildMail` 里没有校验 `MailAttachment.Type` 是否合法、`Amount` 是否 > 0。未来做附件领取时，如果发现历史上存了非法附件（Type 拼错或 Amount 为 0），会导致领取逻辑要额外处理脏数据。现在加校验是零成本，晚做有历史包袱。

**5. sourceType 没有枚举校验**

`buildMail` 对 `sourceType` 只是 `TrimSpace`，不做枚举限制。而 `mailType` 和 `senderType` 都有 `normalize*` 函数兜底到默认值。`sourceType` 的缺失会导致数据库里存在任意字符串，`idx_mails_source` 索引也会失去意义，后续按 `sourceType` 过滤或统计时数据质量很差。

**6. 没有 MarkMailRead 的独立 API 路由**

设计文档里规划了 `POST /api/v1/mails/{mailId}/read` 接口，但 `router.go` 里没有注册这个路由，只有通过 `GetMail`（详情接口）的副作用来标记已读。这意味着前端没有办法只标记已读而不拉取全量内容，也不符合设计文档的约定。OpenAPI 里也没有这个路径，但设计文档有——文档、接口、实现三层都对不上。

**7. Admin 查看信函只能翻第 1 页（前端固定 pageSize=10，无翻页 UI）**

`MailAdminPanel.tsx` 调 `adminApi.getPlayerMails(playerId.trim(), 1, 10)` 写死了第 1 页，面板里没有翻页按钮。玩家信函超过 10 封后，GM 无法查看更多，排查问题时有盲区。

**8. 批量发信结构预留但 Admin 端完全空白**

设计文档明确说"接口方向提前保留"，但 `router.go`、`handler.go`、`service_mail.go`、Admin 前端里没有任何占位或注释，后续做批量邮件时需要全部重新设计，"预留"是假的。

**9. 过期逻辑只存字段，没有任何消费**

`buildMail` 解析了 `expiresAt` 并存入数据库，但 `ListMails`、`GetMail`、`DeleteMail`、`SendMail` 全都不检查是否过期。玩家当前可以打开和操作已过期邮件。未来如果在附件领取里加过期校验，而列表和详情不过滤，会出现"能看到却领不了"的体验问题。

**10. MemoryRepository 的 mail 上限是每玩家 1000 封，MySQL 没有上限**

Memory 版做了 `> 1000` 截断，MySQL 版没有任何上限控制。两套实现行为不同，切换存储时玩家体验会变化。同时，MySQL 版没有定期清理机制，长期运行后 `mails` 表会无限增长。

---

### 总结

核心隐患主要集中在：时间格式一致性、已过期邮件未被过滤、删除返回硬编码分页、独立已读接口缺失。扩展性瓶颈主要在：`sourceType` 无校验导致历史数据脏、附件字段无校验、批量发信预留是空头支票、Admin 缺分页。这些问题现在修复都很小，放到有真实用户数据后修复成本会成倍增加。

## 前端信函模块问题分析（claude 分析，2026-06-11）

> 以下为 claude 对前端信函模块（`web/src/pages/mail/`）的审查结论，与上文后端分析独立。

### 一、严重：类型筛选与服务端分页冲突（功能性 Bug）

`api/game.ts` 的 `listMails` 只传 `playerId / page / pageSize`，**不传类型**，后端返回的是"全部信函"的某一页（10 封）。但 `MailPage.tsx` 第 45-47 行又在客户端用 `filteredMails` 对 `activeType` 做了二次过滤：

```ts
const filteredMails = activeType === 'all' ? mails : mails.filter(m => m.mailType === activeType)
```

这导致一连串错乱：

- **筛选只在当前这 10 封里生效**。选"奖励"时，如果本页 10 封里只有 2 封是奖励，就只显示 2 封，其它页的奖励信函看不到。
- **分页计数对不上**。`totalMails` / `totalPages` 始终是全部信函的总数，不随筛选变化，会出现"显示 2 封却告诉你共 8 页"的矛盾界面。
- **"暂无信函"误导**。翻到某页恰好没有该类型就显示"暂无信函"，但别的页里其实有。

**修复方向**：把类型下推到后端，`listMails` 增加 `mailType` 参数，服务端按类型分页；删掉 `filteredMails` 这层客户端过滤。

### 二、中等：筛选项漏了 system_notice 类型

`data.ts` 的 `TYPE_CONFIG` 定义了 `system_notice`（系统），但 `TYPE_OPTIONS` 筛选标签里没有它。结果"系统"类信函只能在"全部"下看到，无法单独筛选。

### 三、中等：打开信函时多一次冗余请求 + 已读状态三处来源

`handleSelectMail`（第 88-104 行）流程：本地乐观标已读 → `getMail` 拉详情（后端也标已读）→ 再 `loadMails(currentPage)` 整页重拉。每点开一封信都要再拉一整页列表，浪费请求，且"已读"状态有三个来源（本地乐观、详情接口、列表重拉），网络慢时可能计数闪烁或竞态。`getMail` 之后无需再 `loadMails`。

### 四、轻微：UX / 一致性问题

- **复制地址无反馈**（第 129-132 行）：`navigator.clipboard?.writeText` 在非 HTTPS 下为 `undefined` 静默失败，成功也无 toast 提示。
- **发私信成功后不关弹窗、不刷新**（第 134-153 行）：成功只显示文字提示，弹窗不关、列表不刷新。
- **`expiresAt` 渲染未格式化**（MailDetail 第 27 行）：直接渲染原始字符串，与同组件 `createdAt` 用 `toLocaleString` 不一致。
- **附件 key 可能冲突**（MailDetail 第 52 行）：`key={type-itemId}`，同一封信两个相同 type+itemId 附件会撞 key。
- **领取附件后不更新未读数**（第 113-127 行）：只更新 resources/cityGold，未同步 `unreadMailCount`，若后端领取时顺带标已读则前端计数与服务端不一致。

### 小结

核心要修的是**类型筛选目前是"假筛选"**，只在当前页生效，和服务端分页放一起会出现错乱计数和"暂无信函"，需后端支持按类型分页根治。其余为请求冗余和若干 UX/一致性细节。
